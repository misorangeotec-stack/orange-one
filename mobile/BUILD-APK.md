# Building a local Android APK (no Expo/EAS cloud)

This app normally runs in **Expo Go** (`npm start` → scan QR). That still works and
is untouched. This doc is for producing a **standalone `.apk`** you can install on a
phone for testing, built entirely on this machine with Android Studio's toolchain.

## TL;DR — build an APK

```powershell
cd mobile
npm run apk
```

Output: **`mobile\apk-output\orange-one.apk`** (~45 MB, arm64, installs on any modern phone).

Install it on a USB-connected phone (USB debugging on):

```powershell
adb install -r mobile\apk-output\orange-one.apk
```

…or just copy the `.apk` to the phone and tap it (allow "install from unknown sources").

## One-time setup (already done on this machine)

These were done once and don't need repeating:

1. **Android SDK** at `%LOCALAPPDATA%\Android\Sdk` (from Android Studio).
2. **Command-line tools** installed into the SDK (`sdkmanager`).
3. **NDK 27.1.12297006 + CMake 3.22.1 + platform android-36** installed via `sdkmanager`
   (React Native compiles native C++, so these are required).
4. **Windows long paths enabled** — `HKLM\SYSTEM\CurrentControlSet\Control\FileSystem\LongPathsEnabled = 1`.
   *Required.* Without it the native codegen files exceed Windows' 260-char path limit and
   the build fails with `ninja: error: ... Filename longer than 260 characters`.
5. **JDK** = Android Studio's bundled `jbr` (Java 21). The build script points `JAVA_HOME`
   at `C:\Program Files\Android\Android Studio\jbr` automatically — nothing on your PATH.

If you ever move Android Studio or the SDK, edit the two paths at the top of `build-apk.ps1`.

## How it works

`build-apk.ps1`:
1. Sets `JAVA_HOME` / `ANDROID_HOME` for the session.
2. (only with `-Clean`) runs `npx expo prebuild --platform android --clean` to regenerate
   the native `android/` project from `app.json`.
3. Runs `gradlew assembleRelease`.
4. Copies the APK to `apk-output\orange-one.apk`.

The release APK is signed with the **debug keystore** (auto-generated by Expo's template),
so it installs and runs standalone — no Metro/dev server needed. That's fine for testing.
For Play Store distribution you'd generate a real upload keystore and wire it into
`android/app/build.gradle` — not needed for internal testing.

## Speed

- **First build:** slow (~15–35 min) — downloads Gradle, compiles all native code.
- **Everyday builds (JS changes only):** ~3–6 min — native code is cached, only the JS
  bundle + packaging re-run.
- The build compiles **arm64-v8a only** (`android/gradle.properties` →
  `reactNativeArchitectures=arm64-v8a`). That covers all modern real phones and keeps builds
  fast. If you need to run on an **Android emulator**, add `x86_64` back to that line.

## When to use `-Clean`

Run `npm run apk:clean` (regenerates `android/`) after you:
- add/remove a **native module** or Expo config plugin,
- change native config in `app.json` (icons, permissions, package name, plugins),
- bump the Expo SDK / React Native version.

For plain JS/TS changes you do **not** need `-Clean` — a clean rebuild is much slower.

> A clean prebuild overwrites `android/gradle.properties` and `android/local.properties`;
> the script re-applies the arm64-only and SDK-path settings automatically afterward.

## Notes

- The whole `android/` folder is **generated and gitignored** (Expo "Continuous Native
  Generation"). `app.json` is the source of truth. Safe to delete `android/`; `-Clean`
  regenerates it.
- Don't delete `%USERPROFILE%\.gradle` between builds — that's the cache that keeps
  rebuilds fast.
- `apk-output/` is gitignored.
